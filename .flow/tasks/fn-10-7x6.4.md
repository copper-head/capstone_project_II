# fn-10-7x6.4 Frontend JavaScript (SSE reader + terminal rendering + log viewer)

## Description
Implement the client-side JavaScript for both pages. The pipeline page JS handles form submission (multipart POST with file or text + drag-and-drop + file preview + Ctrl+Enter), reads the SSE stream from the POST response body using `fetch()` + `ReadableStream.getReader()`, updates the visual stage tracker in real-time, renders pipeline results as terminal-style output (CRUD actions with reasoning, assumptions, sync status, event matching, memory actions, token/cost footer, summary stats), and populates the expandable log viewer with real-time raw logs. The memory page JS fetches the memory API and renders grouped entries in terminal-style accordions.

**Size:** M
**Files:**
- `src/cal_ai/web/static/app.js` — Pipeline page: form submit handler, drag-and-drop with file preview, Ctrl+Enter shortcut, SSE stream reader (POST-based), stage tracker updater, terminal-style result renderer, log viewer populator, error handling (disconnect, 409, crash)
- `src/cal_ai/web/static/memory.js` — Memory page: fetch `/api/memories`, group by category, render terminal-style accordion sections with entries

## Approach
- **POST-based SSE**: `EventSource` only supports GET. Use `fetch('/api/pipeline/run', {method:'POST', body: formData})` then `response.body.getReader()` with `TextDecoder` to parse SSE events. This is ~30 lines of JS for the SSE parser.
- SSE parser: accumulate chunks, split on `\n\n`, parse `event:` and `data:` fields
- **SSE event types handled**:
  - `event: stage` → update stage tracker (toggle CSS classes: `stage--pending` → `stage--running` → `stage--complete`)
  - `event: log` → append to log viewer container (real-time log streaming)
  - `event: result` → parse JSON, render terminal-style output
  - `event: error` → show error in terminal output area
  - `event: done` → finalize (re-enable submit, mark remaining stages)
- Stage IDs in SSE events: `"1"`, `"1b"`, `"1c"`, `"2"`, `"3"`, `"4"` — these map to the 6 stage tracker DOM elements from Task 3
- **Terminal-style result rendering**: Build terminal output as text lines in the dark output block:
  - Each event: `✅ CREATE: Lunch with Sarah` / `✅ UPDATE: Team Standup (matched: event #3 — Team Standup, Mon 10:00)` / `❌ FAILED: ...`
  - Below each event: readable timestamp, AI reasoning, assumptions inline
  - Sync status: `✅ Synced` / `⏸️ Dry run (not synced)` / `❌ Failed: <reason>`
  - Memory section (always shown): `🧠 MEMORY ADD: [people] lunch_preference = "Italian" (high)` with reasoning
  - Token/cost footer: `Tokens: 1,234 in / 567 out — Est. cost: $0.003`
  - Summary line: `3 events created, 1 updated, 0 failed — 12.4s`
  - Zero events: `No calendar events found in this conversation`
  - Dry-run: banner line `⚠️ DRY RUN — no calendar changes made`
- **Drag-and-drop**: Listen for `dragover`/`drop` events on the drop zone. On drop, read file via `FileReader`, populate the textarea with file contents for preview/editing.
- **File picker**: On file input `change` event, read file via `FileReader`, populate textarea.
- **Ctrl+Enter**: `keydown` listener on textarea, check `e.ctrlKey && e.key === 'Enter'`, trigger form submit.
- **Submit button state**: Change text to "Running..." and disable during pipeline run. Re-enable on `event: done` or error.
- **Error handling**:
  - Fetch failure / SSE disconnect → "Connection lost. Click to retry." message in results area
  - HTTP 409 → "A pipeline is already running. Please wait." inline message
  - Pipeline crash (`event: error`) → error message in terminal output, stage tracker shows where it stopped
- **After completion**: Clear textarea input, results stay visible for review.
- **Memory page**: `fetch('/api/memories')` → response is flat JSON array → group by `category` field client-side → render terminal-style accordion sections with entry rows (key bold, value, confidence badge). Show instructional empty state when array is empty.

## Key context
- `ReadableStream.getReader()` returns `Uint8Array` chunks — use `TextDecoder` for streaming decode
- SSE format: each event is `event: <type>\ndata: <json>\n\n` — split buffer on double newlines
- The submit button should be disabled while pipeline is running (re-enabled on `event: done` or error)
- Stage names to display: "Parsing Transcript" (1), "Loading Memories" (1b), "Fetching Calendar" (1c), "Extracting Events (LLM)" (2), "Syncing Calendar" (3), "Updating Memory" (4)
- **Not all 6 stages are guaranteed**: Stages 1b/1c may not appear if memory/calendar steps are skipped or empty. Pipeline can also early-return for zero utterances. Stage tracker must handle partial progression gracefully (leave unvisited stages as pending).
- **Confidence is categorical** (`"high"`, `"medium"`, `"low"`) — map to CSS classes directly, no numeric thresholds
- Memory API returns flat array — JS must group by `category` field
- **Memory entries**: show key, value, confidence only — skip timestamps and source counts
- **Token/cost**: `result.token_usage` has `prompt_tokens`, `output_tokens`, `estimated_cost_usd`
- **Memory actions**: `result.memory_actions` is array of `{action, category, key, new_value, confidence, reasoning}`
- **File preview**: Both drag-and-drop and file picker should populate the textarea with file contents
- **Readable timestamps**: Format from ISO to readable (e.g., "Wed Mar 4, 12:00 PM – 1:00 PM") in JS
