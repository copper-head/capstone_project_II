# fn-10-7x6.2 Response schemas + API endpoints (memory + pipeline SSE + log streaming)

## Description
Create Pydantic response models that serialize the pipeline's stdlib dataclasses and nested Pydantic models into clean JSON — including memory action details and token/cost data. Implement the pipeline SSE endpoint (`POST /api/pipeline/run`) and memory JSON endpoint (`GET /api/memories`). The SSE endpoint accepts multipart form data (file upload or pasted text), runs the pipeline in a background thread via `asyncio.to_thread()`, captures stage progress via a custom `logging.Handler` with stage synthesis state machine, streams raw log lines as `event: log`, and streams SSE events to the client.

**Size:** M
**Files:**
- `src/cal_ai/web/schemas.py` — Pydantic response models: `EventResponse`, `SyncResultResponse`, `FailedEventResponse`, `MemoryActionResponse`, `TokenUsageResponse`, `PipelineResultResponse`, `MemoryResponse` with `from_pipeline_result()` converter
- `src/cal_ai/web/sse.py` — `PipelineLogCapture(logging.Handler)` with stage state machine + log forwarding + `pipeline_sse_generator()` async generator
- `src/cal_ai/web/routes.py` — expand with `POST /api/pipeline/run` (SSE) and `GET /api/memories`
- `tests/unit/test_web_schemas.py` — schema conversion tests (dataclass→Pydantic, Path serialization, nested models, memory actions, token usage)
- `tests/unit/test_web_app.py` — expand with pipeline SSE mock test, memory endpoint test, log event test

## Approach
- Response models mirror fields from `PipelineResult` (`pipeline.py:78-120`), `ExtractedEvent` (`models/extraction.py:29-60`), `EventSyncResult` (`pipeline.py:39-62`), `FailedEvent` (`pipeline.py:65-75`), `MemoryRecord` (`memory/models.py:22-43`), `MemoryAction` (`memory/models.py:90-115`)
- **New schemas for memory actions and token usage**:
  - `MemoryActionResponse`: mirrors `MemoryAction` fields (action, category, key, value, confidence, reasoning). Excludes `target_memory_id` (internal remapped ID, not useful for display).
  - `TokenUsageResponse`: aggregated token counts from all LLM calls. Fields: `prompt_tokens: int`, `output_tokens: int`, `total_tokens: int`, `estimated_cost_usd: float | None`. Built from `extraction_usage_metadata` + `memory_usage_metadata` by extracting `prompt_token_count` and `candidates_token_count` from each SDK object.
  - Cost calculation: Gemini 2.5 pricing — $1.25/1M input, $10.00/1M output. Handle missing/None usage objects gracefully.
- `from_pipeline_result(result: PipelineResult)` class method handles:
  - `Path` → `str` (for `transcript_path`)
  - `memory_usage_metadata` + `extraction_usage_metadata` → `TokenUsageResponse` (aggregate token counts, compute estimated cost)
  - `memory_actions` → list of `MemoryActionResponse`
  - Nested `ExtractedEvent` (already Pydantic) → `EventResponse`
- **Stage synthesis state machine** in `PipelineLogCapture`:
  - Pipeline only logs `"Stage 1: ..."` through `"Stage 4: ..."` explicitly
  - Sub-stages 1b and 1c do NOT have explicit log lines
  - Handler maps log message patterns to synthetic stage events:
    - `"Stage 1: ..."` → stage 1 running
    - `"Memory context loaded"` or `"Memory load failed"` → stage 1b complete
    - `"Stage 2: ..."` → if 1b/1c not yet emitted, infer them complete (boundary fallback), then stage 2 running
    - `"Calendar context fetched"` or `"Calendar context unavailable"` → stage 1c complete
    - `"Stage 3: ..."` → stage 3 running (stage 2 complete)
    - `"Stage 4: ..."` → stage 4 running (stage 3 complete)
  - **Terminal rule**: On pipeline completion (result available) or `done` event, mark any still-running stage (especially stage 4) as complete before emitting `done`
  - **Empty-memory fallback**: If no memory log line appears before `"Stage 2"`, infer 1b complete at Stage 2 boundary
  - Handler filters by `threading.get_ident()` for thread isolation
  - Pushes to `asyncio.Queue` via `loop.call_soon_threadsafe(queue.put_nowait, ...)`
- **Log forwarding**: The same handler also forwards ALL log messages from `cal_ai.*` loggers as `event: log` SSE events (with `data: {"message": "...", "level": "INFO", "logger": "cal_ai.pipeline"}`). This is separate from stage synthesis — every log line becomes a log event regardless of whether it also triggers a stage transition.
- **Atomic lock acquisition**: Use `asyncio.wait_for(lock.acquire(), timeout=0)` — catches `TimeoutError` → HTTP 409. Release in `try/finally`.
- Pipeline SSE endpoint: validate exactly one of file/text provided, write to `NamedTemporaryFile`, acquire lock atomically, run `asyncio.to_thread(run_pipeline, ...)`, yield SSE events from queue, cleanup temp file in finally block
- **Memory API returns flat list**: `GET /api/memories` returns flat JSON array of `MemoryResponse` objects ordered by category/key. Client-side grouping in `memory.js`. Uses `load_memory_settings()` → `MemoryStore(path)` → `load_all()` → `close()` pattern from `__main__.py:275-315`. Memory entries include key, value, confidence, and category (skip timestamps/source_count per interview decision).
- Use `StreamingResponse(generator, media_type="text/event-stream")` — no extra SSE deps needed
- Test with `from fastapi.testclient import TestClient` (Starlette's TestClient). Mock `run_pipeline` and have mock emit log lines via `logging.getLogger("cal_ai.pipeline")` so SSE parsing is exercised. Use `client.stream(...)` context manager for SSE tests.

## Key context
- `PipelineResult.transcript_path` is a `Path` object — must serialize to string
- `PipelineResult.memory_usage_metadata` is a list of Gemini SDK `usage_metadata` objects with `prompt_token_count`, `candidates_token_count`, `total_token_count`
- `PipelineResult.extraction_usage_metadata` (new from Task 1) — same structure as memory_usage_metadata
- `PipelineResult.memory_actions` (new from Task 1) — list of `MemoryAction` Pydantic models
- The pipeline logger emits `"Stage 1: ..."` through `"Stage 4: ..."` but NOT `"Stage 1b"` or `"Stage 1c"` — synthesis is required
- Memory-related log lines: `"Memory context loaded..."`, `"Memory load failed..."` (see `pipeline.py:180-195`). Note: there is NO `"No memories found"` line — empty-memory case has no explicit log, so infer 1b complete at Stage 2 boundary
- Calendar-related log lines: `"Calendar context fetched..."`, `"Calendar context unavailable..."` (see `pipeline.py:197-220`)
- `asyncio.Lock` has no atomic try-acquire — must use `wait_for(..., timeout=0)` pattern
- `from fastapi.testclient import TestClient` is the correct import (Starlette's, backed by httpx)
- Stage 4 has no following `"Stage 5"` log — must use terminal rule to mark it complete on pipeline finish
- `MemoryResponse` should include: `category`, `key`, `value`, `confidence` — skip `id`, `source_count`, `created_at`, `updated_at` per interview
