# fn-10-7x6.3 Frontend templates + CSS styling (split view, terminal aesthetic)

## Description
Create the Jinja2 HTML templates and CSS for both pages with a split-view layout and terminal-style output aesthetic. The base template defines shared structure (nav bar with "Cal-AI" text, config warning banner, CSS link, JS block). The pipeline page (`index.html`) has a split-view layout: left panel with input form (tab toggle between file upload with drag-and-drop zone and text paste), right panel with visual stage tracker (vertical stepper) and terminal-style output area plus expandable log viewer. The memory page (`memory.html`) displays memories in terminal-style accordion sections grouped by category. Add page-serving routes (`GET /`, `GET /memory`).

**Size:** M
**Files:**
- `src/cal_ai/web/templates/base.html` — HTML5 base layout: nav bar with "Cal-AI" text + page links, config warning banner (conditional on `config_warnings`), CSS link, content block, JS block
- `src/cal_ai/web/templates/index.html` — Pipeline page: split-view layout (left: input section with upload/paste tabs, dry-run checkbox, submit button; right: stage tracker vertical stepper + terminal output block + expandable log viewer)
- `src/cal_ai/web/templates/memory.html` — Memory viewer: terminal-style accordion sections by category, entry rows (key, value, confidence badge), instructional empty state
- `src/cal_ai/web/static/style.css` — All styling: CSS custom properties (color scheme), split-view layout (CSS grid or flexbox, desktop only), terminal output block (monospace, dark background), stage tracker (pending/running/complete states with spinner), confidence badges (green/yellow/red mapped from categorical high/medium/low), action badges (CREATE=blue, UPDATE=orange, DELETE=red, SKIP=gray), sync status indicators (✅/⏸️/❌), tab toggle for input modes, drag-and-drop zone styling, expandable log viewer (collapsed by default), accordion styling for memory categories, config warning banner (yellow), dry-run indicator banner
- `src/cal_ai/web/routes.py` — add `GET /` and `GET /memory` page routes returning `TemplateResponse`, pass `config_warnings` to template context
- `tests/unit/test_web_app.py` — expand with page route tests (200 status, content-type text/html, config warning rendering)

## Approach
- Reference `demo_output.py:32-58` (`format_pipeline_result()`) for what data each section displays
- **Split-view layout**: CSS grid or flexbox with two columns — left panel (~40% width for input), right panel (~60% width for results). Desktop only, no responsive breakpoints needed.
- **Terminal output block**: `<pre>` or `<div>` with monospace font, dark background (#1e1e2e or similar), light text (#cdd6f4), subtle border. Content populated by JS (Task 4).
- **Stage tracker stages** (6 total, matching synthesized SSE events from Task 2):
  1. "Parsing Transcript" (Stage 1)
  2. "Loading Memories" (Stage 1b — synthesized)
  3. "Fetching Calendar" (Stage 1c — synthesized)
  4. "Extracting Events (LLM)" (Stage 2)
  5. "Syncing Calendar" (Stage 3)
  6. "Updating Memory" (Stage 4)
- **Config warning banner**: Base template checks `config_warnings` template variable. If non-empty, renders a yellow banner at top of page with warning messages. `routes.py` passes `request.app.state.config_warnings` to all `TemplateResponse` calls.
- **Drag-and-drop zone**: A styled area in the file upload tab with dashed border, hover/active states. File picker button inside the zone.
- **Expandable log viewer**: A `<details>/<summary>` element or custom CSS toggle below the terminal output. Collapsed by default. Contains a scrollable terminal-style block for raw log lines.
- **Memory page accordion**: Each category is a collapsible section. Entries show key (bold), value, and confidence badge. Uses terminal-style monospace presentation.
- **Memory empty state**: Centered instructional text: "No memories yet. Run a pipeline to start building memory about the calendar owner."
- Templates use Jinja2 block inheritance: `{% extends "base.html" %}` with `{% block content %}` and `{% block scripts %}`
- All interactive behavior is in JS (Task 4) — templates provide the DOM structure with `id` attributes for JS to target
- No JavaScript in this task — just HTML structure + CSS styling + static rendering
- Remove `.keep` placeholder files from `templates/` and `static/` (created in Task 1) once real files are added

## Key context
- Pipeline page results section is hidden by default (CSS `display: none`), shown by JS after pipeline completes
- Stage tracker items need distinct CSS classes for states: `stage--pending`, `stage--running` (spinner), `stage--complete` (checkmark)
- Stages 1b and 1c are synthesized by the SSE handler (Task 2), not actual pipeline log stages
- **Confidence is categorical** (`"high"`, `"medium"`, `"low"`), NOT numeric — badge CSS maps: `high` = green, `medium` = yellow, `low` = red
- Memory categories from the system: preferences, people, vocabulary, patterns, corrections
- **Desktop only** — no mobile/tablet responsive breakpoints needed
- **Memory entries show key + value + confidence only** — no timestamps, no source counts
- **Config warnings** are stored in `app.state.config_warnings` (set in Task 1's `create_app()`)
