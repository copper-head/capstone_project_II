# Web Frontend (FastAPI + Vanilla HTML/JS)

## Overview

Cal-AI is currently CLI-only. The pipeline produces rich structured data (extracted events with AI reasoning, calendar sync results, memory updates) but it's only viewable as console text. Adding a web frontend makes the demo significantly easier to show off â€” the audience sees pipeline stages animate in real-time via SSE, browses extracted events with confidence badges, and explores the memory system through a dedicated viewer.

**Approach**: FastAPI REST backend serving Jinja2 templates with vanilla JS. No Node.js, no SPA framework. Server-Sent Events (SSE) stream pipeline progress to the browser in real-time. The existing CLI is untouched â€” the web layer wraps `run_pipeline()` without modifying it.

## Scope

**In scope:**
- `python -m cal_ai serve` subcommand (FastAPI + uvicorn)
- Pipeline page: transcript upload/paste â†’ SSE stage progress â†’ results display
- Memory viewer page: browse stored memories grouped by category
- Dry-run checkbox support
- Docker configuration for web server mode
- Unit tests for web layer
- Small `PipelineResult` extension: carry memory actions and extraction token usage (needed for web display)

**Out of scope:**
- Authentication / multi-user
- Run history / persistence of past results
- Memory editing (write operations) through the web UI
- Mobile/tablet responsive layout (desktop only)

## Key Decisions (from interview)

### Audience & Style
- **Audience**: Professor/graders only â€” functionality and clarity over polish
- **Visual style**: Clean & minimal base, but results use a **terminal-style aesthetic** (monospace font, dark background) â€” consistent across both pages
- **Branding**: Just "Cal-AI" text in the nav bar â€” no logo, no tagline

### Layout & Navigation
- **Split view layout**: Input panel on left, results panel on right (side-by-side)
- **Navigation**: Two pages only â€” Pipeline and Memory
- **Stage tracker**: Visual stepper (UI element with dots/spinners/checkmarks), positioned wherever fits cleanest in the results panel
- **Web UI is the primary demo vehicle** â€” CLI remains as fallback but the web UI is what gets shown

### Pipeline Page â€” Input
- **Both input methods**: File upload and text paste equally important, tab toggle between them
- **File upload**: Drag-and-drop zone + file picker button
- **File preview**: When a file is uploaded, its contents populate the textarea so the user can review/edit before submitting
- **No file size limit** â€” it's a demo, accept whatever
- **Textarea**: Fixed reasonable size (~10-15 lines), scrolls for longer content
- **Keyboard shortcut**: Ctrl+Enter submits the form
- **Dry-run checkbox** with clear indicator banner when active

### Pipeline Page â€” Results (Terminal-Style Output)
- **Terminal aesthetic**: Monospace font, dark background block â€” looks like watching the pipeline run in a terminal
- **Event display**: Each event shows:
  - CRUD action (CREATE/UPDATE/DELETE) with clear sync status (âś… Synced, âŹ¸ď¸Ź Dry run, âťŚ Failed: reason)
  - Title and readable timestamps (e.g., "Wed Mar 4, 12:00 PM â€“ 1:00 PM")
  - AI reasoning for the action
  - Assumptions shown inline (e.g., "Assumed: duration 1 hour (not specified)")
  - For UPDATE/DELETE: show which existing calendar event was matched
- **Memory updates always shown** in the terminal output, even when zero events are extracted
  - Full detail: action (ADD/UPDATE/DELETE), category, key, value, confidence, reasoning
  - Data source: `MemoryAction` objects from `MemoryWriteResult.actions`, carried through `PipelineResult.memory_actions`
- **Token/cost info**: Footer of the terminal output (tokens in/out + estimated cost)
  - Data source: `ExtractionResult.usage_metadata` (extraction call) + `MemoryWriteResult.usage_metadata` (memory calls). Both contain Gemini SDK `usage_metadata` with `prompt_token_count`, `candidates_token_count`, `total_token_count`
  - Cost calculation: Gemini 2.5 pricing ($1.25/1M input, $10.00/1M output)
  - `PipelineResult` extended with `extraction_usage_metadata` (new) alongside existing `memory_usage_metadata`
- **Summary stats**: Bottom of terminal output (event counts, warnings, duration)
- **Zero events**: Clear message "No calendar events found in this conversation" + memory updates still shown
- **Partial failure**: Failed events shown inline with âťŚ indicator and error message

### Pipeline Page â€” Expandable Log Viewer
- **Real-time log streaming**: Raw pipeline logs streamed via SSE as `event: log` events alongside stage events
- **Logger scope**: Capture all `cal_ai.*` loggers (not just `cal_ai.pipeline`)
- **Position**: Collapsed section below the terminal output in the right panel
- **Purpose**: Lets the professor dig into raw pipeline logs if they want to see more detail

### Pipeline Page â€” State Management
- **Submit button**: Text changes during run (e.g., "Run Pipeline" â†’ "Running...")
- **Concurrent runs**: Inline message "A pipeline is already running. Please wait." near the submit button
- **After completion**: Input clears, results stay visible â€” user can paste a new transcript immediately
- **SSE disconnect**: Show "Connection lost" error + manual retry option
- **Pipeline crash**: Error shown in the terminal output area, stage tracker shows where it stopped
- **Pipeline speed**: 10-30 seconds typical â€” stage progress is important feedback

### Pipeline Page â€” Dry-Run
- **Clear indicator**: Banner or badge saying "DRY RUN â€” no calendar changes made"
- **Sync status**: Shows "âŹ¸ď¸Ź Dry run (not synced)" vs "âś… Synced" â€” distinct from live results

### Missing Configuration Warning
- **Startup-time config check**: On app startup, check for `GEMINI_API_KEY` env var. If missing, set a flag on the app instance
- **UI banner**: If config flag is set, base template renders a warning banner: "Warning: GEMINI_API_KEY not configured. Pipeline runs will fail."

### Memory Page
- **Terminal aesthetic**: Consistent with pipeline page
- **Grouping**: Accordion sections by category (preferences, people, vocabulary, patterns, corrections)
- **Entry detail**: Key + value + confidence indicator per memory entry (skip timestamps, source counts)

- **Refresh behavior**: Always loads fresh data on page load (auto-refresh)

### Browser & Screen
- **Desktop only** â€” no mobile/tablet responsive needed
- **Browser support**: Chrome + Firefox

## Pipeline Data Model Extension

The web frontend needs two fields that `PipelineResult` currently doesn't carry:

1. **`memory_actions: list[MemoryAction]`** â€” The per-action memory details (action, category, key, value, confidence, reasoning). Already available on `MemoryWriteResult.actions` but not stored on `PipelineResult`. Pipeline change: `result.memory_actions = write_result.actions`.

2. **`extraction_usage_metadata: list`** â€” Token usage from the extraction LLM call. Already available on `ExtractionResult.usage_metadata` but not stored on `PipelineResult`. Pipeline change: `result.extraction_usage_metadata = extraction.usage_metadata`.

These are additive-only changes to `PipelineResult` (two new fields with default empty lists). No existing behavior changes.

## Architecture

```
Browser                    FastAPI                     Pipeline
  |                          |                            |
  |-- POST /api/pipeline/run -->                          |
  |   (multipart: file or text)                           |
  |                          |-- write temp file -------->|
  |                          |-- asyncio.to_thread ------>|
  |                          |   (run_pipeline)           |
  |<-- SSE: stage events ----|<-- logging.Handler --------|
  |<-- SSE: log events ------|<-- logging.Handler --------|
  |<-- SSE: result ----------|<-- PipelineResult ---------|
  |                          |-- cleanup temp file        |
  |                          |                            |
  |-- GET /api/memories ---->|                            |
  |<-- JSON: memories -------|-- MemoryStore.load_all() --|
```

**SSE via log capture**: A custom `logging.Handler` on `cal_ai` root logger pushes messages to an `asyncio.Queue`. The handler filters by thread ID (ensuring isolation). The SSE generator drains the queue via `asyncio.wait_for()` while `run_pipeline()` runs in a background thread via `asyncio.to_thread()`. Zero pipeline code changes (beyond the two additive `PipelineResult` fields).

**Stage event synthesis**: The pipeline logger emits `"Stage 1: ..."` through `"Stage 4: ..."` but does NOT emit `"Stage 1b"` or `"Stage 1c"` lines. The SSE handler synthesizes sub-stage events by matching related log messages:
- Stage 1b (Loading Memories): triggered by `"Memory context loaded"` or `"Memory load failed"` log lines
- Stage 1c (Fetching Calendar): triggered by `"Calendar context fetched"` or `"Calendar context unavailable"` log lines
- **Boundary fallback**: If no 1b/1c log lines appear before `"Stage 2"`, infer them complete at the Stage 2 boundary
- **Terminal rule**: On pipeline completion (result available), mark any still-running stage (especially Stage 4) as complete before emitting `done`. Stage 4 has no following `"Stage 5"` log to trigger its completion.

**Log streaming**: In addition to synthesized stage events, raw pipeline log lines are forwarded as `event: log` SSE events for the expandable log viewer. The log handler captures all messages from `cal_ai.*` loggers (filtering by `name.startswith("cal_ai")`).

**POST-based SSE**: Since `EventSource` only supports GET, the JS client uses `fetch()` + `ReadableStream.getReader()` to parse SSE events from the POST response body.

**Concurrency guard**: An `asyncio.Lock` prevents overlapping pipeline runs. Acquisition uses `asyncio.wait_for(lock.acquire(), timeout=0)` which raises `TimeoutError` â†’ HTTP 409. The lock is released in a `try/finally` block. This atomic pattern avoids the race condition of `if lock.locked(): raise`.

## Key design decisions

- **Temp files for transcript input**: `run_pipeline()` requires a `Path`. Write uploaded/pasted text to a named temp file, pass the path, clean up after. Avoids modifying the core pipeline signature.
- **Raw StreamingResponse**: Use FastAPI's built-in `StreamingResponse` with `media_type="text/event-stream"`. Keeps deps minimal â€” no sse-starlette needed.
- **Atomic lock acquisition**: `asyncio.wait_for(lock.acquire(), timeout=0)` with `try/finally` release. Avoids racy `if locked()` check.
- **Dependencies in main deps** (not optional extra): This is a demo product â€” simplicity over flexibility.
- **Pydantic response models**: Mirror `PipelineResult`/`ExtractedEvent`/`EventSyncResult` as Pydantic models with a `from_pipeline_result()` converter. Handles `Path` â†’ `str`, `memory_usage_metadata` â†’ token counts (extract `prompt_token_count`/`candidates_token_count` from SDK objects), nested `ExtractedEvent` (already Pydantic) â†’ `EventResponse`, `memory_actions` â†’ `MemoryActionResponse` list.
- **Memory API returns flat list**: `GET /api/memories` returns a flat JSON array ordered by category/key. Grouping by category is done client-side in `memory.js`. This matches `MemoryStore.load_all()` which already orders by category/key.
- **Stage synthesis in SSE handler**: Rather than modifying pipeline code, the SSE handler maps log message patterns to synthetic stage events via a state machine. Includes boundary fallback (infer 1b/1c at Stage 2 if no explicit logs) and terminal rule (mark Stage 4 complete on pipeline finish).
- **Uvicorn logging**: `_handle_serve()` passes `log_level` and `log_config=None` to `uvicorn.run()` to preserve the existing `setup_logging()` configuration and honour CLI `-v`.
- **Confidence is categorical**: `ExtractedEvent.confidence` is `"high"|"medium"|"low"` (not numeric). CSS maps these directly to badge colors: high=green, medium=yellow, low=red.
- **Docker CMD-only**: Remove ENTRYPOINT, use CMD-only pattern. Default: serve. CLI override: `docker run cal-ai python -m cal_ai run samples/file.txt`.
- **Terminal-style output**: Results display in monospace dark blocks rather than UI cards â€” clearer for academic demo, shows raw pipeline intelligence.
- **Split view**: Input stays visible alongside results â€” user can see transcript context while reviewing what the AI extracted.
- **File preview in textarea**: Drag-and-drop or file picker populates the textarea with file contents for review/edit before submission.
- **Real-time log streaming**: Raw pipeline logs streamed via SSE (`event: log`) alongside stage events, displayed in an expandable section below results.
- **Always show memory section**: Memory updates appear in output even when zero calendar events are extracted. Data from `PipelineResult.memory_actions` (new field).
- **Token/cost footer**: Extract `prompt_token_count` and `candidates_token_count` from Gemini SDK `usage_metadata` objects (both extraction and memory calls). Aggregate totals + estimated cost using Gemini 2.5 pricing.
- **Config warning banner**: App factory checks for `GEMINI_API_KEY` on startup, stores warnings in `app.state.config_warnings`. Base template conditionally renders a yellow warning banner.

## Edge Cases

- **Empty transcript**: Pipeline extracts zero events â†’ "No calendar events found" message + memory section still shown
- **Partial sync failure**: Some events sync, others fail â†’ failed events shown inline with âťŚ and error message
- **Pipeline crash**: Exception â†’ error shown in terminal output area, stage tracker shows where it stopped
- **SSE disconnect**: Connection drops â†’ "Connection lost" message + manual retry option
- **Missing config**: Server starts but shows warning banner for missing API key / calendar credentials
- **Concurrent runs**: Second submit â†’ inline message "A pipeline is already running. Please wait."
- **Empty memory DB**: Memory page shows instructional empty state message
- **No memory log lines**: If no "Memory context loaded" log appears before Stage 2, infer 1b complete at boundary
- **Stage 4 terminal**: Stage 4 has no following stage â†’ mark complete on pipeline finish via terminal rule
- **Large file upload**: No size limit enforced â€” accept whatever the user uploads (demo context)
- **File + text both provided**: Validation error â€” require exactly one input method
- **Dry-run + memory**: In dry-run mode, pipeline skips memory write â†’ memory section shows "Memory write skipped (dry-run mode)"
- **Usage metadata unavailable**: If Gemini SDK doesn't return usage_metadata (e.g., API error), token/cost footer shows "Token usage unavailable"
